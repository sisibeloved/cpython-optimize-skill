# FAQ

## 我现在只想看功能，不看性能

优先：
- `cinderx-test`
- 单个 `run_benchmark.py`
- 先开 HIR dump

不要直接拿开启大量 dump 的结果当性能数据。

## 我只想复现一个 crash

优先：
- 单个 `run_benchmark.py --worker ...`
- 调试线 `cinderx-test`
- 先拿 `jit.log` / HIR，再决定是否进 `gdb`

## 为什么 `python -m pyperformance run` 和单 benchmark 结果不一样

优先怀疑：
- `driver / manager / worker` 进程模型
- `bench_command()` 外部子进程
- worker 环境变量继承
- `sitecustomize` / hook 是否真的命中
- 分析阶段是否换成了另一条命令，导致 HIR 与真实测试不一致

## 什么时候该回到 Kunpeng 宿主机

满足任意一条：
- Docker 和真实环境行为不一致
- 需要最终兼容性复核
- 容器内问题已经缩小到 native 层或系统差异

## SSH 上去以后为什么不能默认直接在裸机上干活

因为 SSH 只解决“连接到机器”，不解决“环境隔离”。

默认流程应该是：
- 先确认独立宿主机目录
- 再进入 Docker 容器
- 再在容器里做主工作流

只有最终系统差异复核、容器无法覆盖的问题，才优先回到裸机。

## 为什么要先强调项目隔离性

因为 Docker 需要 bind mount。

如果宿主机目录本身没有隔离好，就会出现：
- 覆盖已有目录
- 多个 Agent 共享写入
- 把旧实验结果误挂进新容器

所以顺序一定是：
- 先隔离宿主机目录
- 再做 Docker 容器隔离

## 远程代码同步为什么优先 `rsync`

因为它更适合：
- 大仓库
- 频繁迭代
- 保持宿主机目录与本地代码仓一致

如果远端没有 `rsync`，先提醒安装，再开始主工作流。

## 为什么 dump HIR 要优先复用真实命令

因为如果测试阶段和分析阶段用了两条不同命令，常见后果是：
- worker 形态不同
- 子进程继承不同
- 触发的 JIT 路径不同
- 最终看到的 HIR 和真实性能问题并不是同一件事

更稳的做法是：
- 先固定真实命令
- 再只增减 debug 环境变量

## 性能口径为什么要先定义清楚

因为下面四种不是一回事：
- `CPython 解释执行`
- `CPython JIT`
- `CinderX 解释执行`
- `CinderX JIT`

如果口径不清，后面的速度对比和 HIR 分析都容易跑偏。

## 什么时候该写文档，而不是继续试

满足任意一条：
- 已经修掉真实环境 crash
- 已经定位到性能退化根因
- 改了 benchmark 入口、环境变量或护栏约定

## Docker 调试时为什么不建议反复 `docker exec`

因为这会让：
- 调试命令不一致
- 上下文丢失
- 容器内临时状态和日志更难追踪

更推荐：
- 先进入一个固定的长连接交互 shell
- 后续在这个 shell 里持续执行调试命令
- 只有自动化脚本或一次性检查才用 `docker exec`
